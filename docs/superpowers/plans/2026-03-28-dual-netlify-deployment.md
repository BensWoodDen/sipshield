# Dual Netlify Deployment Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Set up two Netlify sites (dev + prod) deploying from `dev` and `main` branches with isolated Stripe and Supabase environments.

**Architecture:** Two independent Netlify sites on the free tier, same GitHub repo. Dev site deploys from `dev` branch with Stripe test keys and Supabase `dev` schema. Prod site deploys from `main` branch with live keys and `public` schema. No code changes required - existing `lib/supabase.ts` schema-switching logic handles everything via env vars.

**Tech Stack:** Netlify CLI (`pnpm netlify`), Git branching, Stripe Dashboard (webhook endpoints), DNS (custom domains)

**Spec:** `docs/superpowers/specs/2026-03-28-dual-netlify-deployment-design.md`

### Task 4: Set environment variables on dev site

**Files:** None (Netlify env vars)

- [ ] **Step 1: Set shared env vars on dev site**

```bash

pnpm netlify env:set NEXT_PUBLIC_SANITY_DATASET "production" --force
pnpm netlify env:set SANITY_API_TOKEN "<value from .env.local>" --force
pnpm netlify env:set SUPABASE_URL "<value from .env.local>" --force
pnpm netlify env:set SUPABASE_SERVICE_ROLE_KEY "<value from .env.local>" --force
pnpm netlify env:set AUTH_PEPPER "<value from .env.local>" --force
pnpm netlify env:set SESSION_SECRET "<value from .env.local>" --force
pnpm netlify env:set RESEND_API_KEY "<value from .env.local>" --force
```

> **Note:** `STRIPE_WEBHOOK_SECRET` and `SANITY_STUDIO_NETLIFY_BUILD_HOOK` are set in later tasks after creating those resources.

### Task 5: Set environment variables on prod site

**Files:** None (Netlify env vars)

Since the CLI is linked to the dev site, use the `--id` flag or temporarily re-link. The simplest approach is to set prod env vars via the Netlify Dashboard.

- [ ] **Step 1: Set env vars on prod site via Netlify Dashboard**

Go to sipshield-prod > **Site configuration > Environment variables** and add:

**Shared (same values as dev):**
- `NEXT_PUBLIC_SANITY_PROJECT_ID`
- `NEXT_PUBLIC_SANITY_DATASET` = `production`
- `SANITY_API_TOKEN`
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `AUTH_PEPPER`
- `SESSION_SECRET`
- `RESEND_API_KEY`

**Prod-specific:**
- `NEXT_PUBLIC_SITE_URL` = `https://sipshield.co.uk`
- `STRIPE_SECRET_KEY` = Stripe LIVE secret key
- `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` = Stripe LIVE publishable key

> **Do NOT set `SUPABASE_SCHEMA`** on prod. Omitting it lets `lib/supabase.ts` default to `public` via NODE_ENV.

### Task 6: Configure custom domains

**Files:** None (DNS + Netlify Dashboard)

- [ ] **Step 1: Add `dev.sipshield.co.uk` to the dev site**

In Netlify Dashboard for sipshield-dev:
1. Go to **Domain management > Add a domain**
2. Enter `dev.sipshield.co.uk`

- [ ] **Step 2: Add `sipshield.co.uk` to the prod site**

In Netlify Dashboard for sipshield-prod:
1. Go to **Domain management > Add a domain**
2. Enter `sipshield.co.uk`

- [ ] **Step 3: Configure DNS records**

At your domain registrar, add:
- `dev` CNAME record pointing to `sipshield-dev.netlify.app`
- Root domain (`sipshield.co.uk`) - either ALIAS/ANAME to `sipshield-prod.netlify.app` or use Netlify DNS

> **Note:** Netlify provides free SSL via Let's Encrypt. Certificates are provisioned automatically once DNS propagates.

- [ ] **Step 4: Verify HTTPS**

After DNS propagation (can take up to 48h, usually minutes):
- Visit `https://dev.sipshield.co.uk` - should load (or show build error if first deploy hasn't happened)
- Check Netlify Dashboard shows SSL certificate as active

### Task 8: Create Netlify build hooks

**Files:** None (Netlify Dashboard + env vars)

Build hooks are used by the Sanity Studio "Rebuild Site" action (`sanity/actions/rebuild-site.ts`).

- [ ] **Step 1: Create build hook for dev site**

In Netlify Dashboard for sipshield-dev:
1. Go to **Site configuration > Build & deploy > Build hooks**
2. Click **Add build hook**
3. Name: `sanity-rebuild`
4. Branch: `dev`
5. Copy the hook URL

- [ ] **Step 2: Set the build hook env var on dev site**

```bash
pnpm netlify env:set SANITY_STUDIO_NETLIFY_BUILD_HOOK "<hook URL from step 1>" --force
```

- [ ] **Step 3: Create build hook for prod site**

In Netlify Dashboard for sipshield-prod:
1. Same process - **Build hooks > Add build hook**
2. Name: `sanity-rebuild`
3. Branch: `main`
4. Copy the hook URL

- [ ] **Step 4: Set the build hook env var on prod site**

Via Netlify Dashboard for sipshield-prod:
- Add `SANITY_STUDIO_NETLIFY_BUILD_HOOK` with the prod hook URL

> **Decision point:** The Sanity Studio "Rebuild Site" action currently uses a single `SANITY_STUDIO_NETLIFY_BUILD_HOOK` env var. Since Sanity Studio runs client-side and uses `SANITY_STUDIO_` prefixed vars, both hooks can't be active simultaneously in one Studio instance. For now, the Studio rebuild button triggers whichever site's hook is configured. If Ben needs to rebuild both, this can be revisited later.

```markdown
## Deployment

Two Netlify sites deploy from different branches:

| | Dev | Prod |
|---|---|---|
| **Site** | sipshield-dev | sipshield-prod |
| **Branch** | `dev` | `main` |
| **Domain** | `dev.sipshield.co.uk` | `sipshield.co.uk` |
| **Stripe** | Test keys | Live keys |
| **Supabase schema** | `dev` (via `SUPABASE_SCHEMA=dev`) | `public` (default) |

### Workflow
- All development happens on `dev` branch. Push triggers dev site rebuild.
- To deploy to production: merge `dev` into `main`.
- Local CLI is linked to the dev site (`pnpm netlify status` to verify).

### Environment variables
- Shared vars (Sanity, Supabase, auth) have the same values on both sites.
- Per-site vars (Stripe keys, site URL, webhook secret, build hook) differ.
- **Never set `SUPABASE_SCHEMA` on prod** - omitting it defaults to `public` via NODE_ENV.
- Env vars are managed via `pnpm netlify env:set` (dev site) or Netlify Dashboard (prod site).
```
